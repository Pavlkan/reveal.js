KISS principle

by Pavel Karman

What is KISS?

Keep it simple, silly

KISS is a design principle which comes to us from the U.S. Navy in 1960

Main idea of KISS

Most systems work best if they are kept simple rather than made complicated; therefore, simplicity should be a key goal in design, and unnecessary complexity should be avoided.

How can we achieve simple and clean code?

1. Simple, but not primitive

We should use adequate tool for our current task

2. Not excessively shortened

Code size should be as long as it needs for understanding.

Couple of words about variables and functions naming

  • One-symbol names: 
    
									const a;
									var y;
  • One-symbol names in certain places like cycles: 
    
									for (let i = 0; i <= someVarName.length; i++) {
										//some code...
									}
  • Unique abbreviations: 
    
									let ua; //userAgent
									let brsr; //browser
  • Widely used abbreviations: 
    
									let btn; //button
									let arr; //array
									let obj; //object
  • Abstract name that doesnâ€™t refer to anything specific: 
    
									let value;

  • Specific name: 
    
									let classNameData;

  • Identical names provide confuse: 
    
									let sameName = {...}
									//...
									if (someCondition) {
										let sameName = 42341;
									}
  • Add unique semantic parts instead: 
    
									let oneName = {...}
									//...
									if (someCondition) {
										let otherName = 42341;
									}
  • Synonymic parts in variables and functions: 
    
									renderSomeCode();
									PaintSomeOtherCode();
									DisplaySomeMoreCode();
  • Chose one main naming style and unificate all names according to their purpose: 
    
									renderCodeA();
									renderCodeB();
  • Not verb name for function: 
    
									function validation() {};
									function F() {}
  • Verb names: 
    
									isValidate(); //is- / are- for predicates 
									showMessage();
									calculateSomeAlgorithm();

DRY principle

Do not repeat yourself

In simple code we do not have repeats, they complicate code, make it unnecessarily big and hard to fix and refactor.

More we have repeats - bigger amounts of code to refactor, bigger number of potential mistakes and bugs

YAGNI principle

You ain't gonna need it

In continuation of the topic of simplifying and reducing code we come to escaping of unused functionality

Is unused functionality bad?

Well, it is not moving you to the task accomplishment. Instead, just like with code duplication, the size of your code increases, as increases the amount of potential mistakes and work.
